# Atlas — going deeper into the machine

*A side adventure, and the most educational stretch of the whole project. Everywhere else on this repo I treated the inference engine as a box that serves tokens. Atlas is where I opened the box, in the language it's actually written in, and got to see the LLM principles I'd only read about show up as code with measurable costs.*

← [back to the overview](README.md)

## What Atlas is, and why I went in

Atlas is an open-source inference engine written in **Rust** — not a Python layer over someone else's CUDA, but the scheduler, the KV cache, the collectives, and the model-specific decode kernels, all the way down. I'd spent this whole project driving engines from the outside (vLLM, Ollama, a runtime bench or two). Atlas was the chance to work from the inside, in a systems language I hadn't used at this depth, on a model big enough that every decision has a number attached.

The model was a 122B sparse-MoE — about 10B active parameters per token, 256 experts, and a hybrid layer stack: 48 layers, three-quarters of them **state-space (linear-attention) layers** and one in four a **full-attention** layer. So in one codebase I got to read all three of the shapes the field argues about — softmax attention, linear/SSM attention, and mixture-of-experts routing — sitting next to each other as functions I could instrument and change.

It didn't fit on one box. The weights plus the fixed MoE scratch buffers overflow a single 128 GB unified-memory board so badly that the KV cache gets *zero* allocatable blocks — I confirmed that by shrinking batch and sequence length to the floor and watching it still fail. That's *why* it ran with **expert parallelism across two boxes** (weights sharded to ~38 GB per rank). Learning #1, before I'd changed a line: "does it fit" is not one number, it's weights + scratch + KV, and the scratch for a 256-expert MoE is not a rounding error.

## The principles, as code

**MoE dispatch is a kernel-selection problem, and my intuition was backwards.** The obvious move for routing tokens to experts is a generic grouped-GEMM — the same kernel that's fast when you're processing a big prefill batch. At decode time, with one token per expert across 36 layers, that kernel is a *net loss*: the sort/permute/pointer-table bookkeeping costs more than the matmul it enables. The thing that actually helped was a set of hand-fused small-batch kernels (a dedicated path for two or three concurrent sequences), which cut the state-space decode step by 15–20%. Above that concurrency there's no fused kernel at all, so the right answer flips back to a plain per-token loop. I revived a dead code path to wire this up correctly and it shipped — net *negative* lines of code, because the fix was mostly deleting a wrong abstraction. Learning: "use the batched kernel" is not a universal truth; at decode with tiny per-expert batches it's the trap.

**Quantization has a principled boundary, not just a size dial.** The weights are 4-bit (NVFP4) packed, with a BF16 remainder that the runtime re-quantizes at load wherever it's safe — the LM head, the speculative head, the MoE experts. There's exactly one place it deliberately *won't*: the state-space mixer's input projections, and specifically the decay-gate parameters, which are forced all the way up to FP32. The reason is beautiful and I'd never have internalized it from a blog post — the recurrent path amplifies error *exponentially* in the decay gate at long context, so a cheaper representation there doesn't cost you a little quality, it costs you coherence at 8k+ tokens. The model is already sitting right at the safe quantization frontier; the last remaining bandwidth win is on the wrong side of a quality cliff, on purpose.

**Speculative decoding is why the single-stream number is good — and the logs lie about it.** A multi-token-prediction head drafts ahead and the main model verifies; acceptance was about 77%, which is ~1.77 tokens per step, which is the whole reason single-stream throughput nearly doubles. The lesson wasn't the mechanism, it was the debugging: the scheduler's log lines *looked* like ~6% acceptance because they're throttled and reject-biased, while the actual Prometheus counter said 77%. Trust the metric, not the narration — which is the exact same lesson as everywhere else on this repo, just one layer down.

**CUDA graphs taught me to find the bottleneck before optimizing it.** Graphs were disabled under expert parallelism, and I assumed that was a conservative gate rather than a fundamental limit — so I prototyped turning them on, and it *worked*: the whole 48-layer decode, including the cross-node collective, captured into a graph cleanly, no hang, coherent output. It also gave exactly zero speedup. Because this model's decode is **memory-bandwidth and network bound, not CPU-launch bound**, and all a graph does is remove launch overhead. The oft-quoted "2× from CUDA graphs" is real — in launch-bound regimes, which means smaller models or single-host. Not here. A whole line of optimization, correctly implemented, measured, and closed as a dead end. That's a good day, even though nothing got faster.

**Parallelism has a ceiling that batching can't beat.** Fully tuned, the 122B did ~35–40 tokens/sec single-stream across the two boxes. The number that taught me the most was the concurrent one: aggregate throughput is *flat* as you add sequences — roughly 35 at one stream, 33 at two, 29 at four. It doesn't scale, because the bottleneck is memory bandwidth plus the cross-node collective over the network, and adding batch doesn't buy you out of a bandwidth wall. For contrast, the same engine on a smaller 80B model on a *single* box hits ~99 tok/s. Going from 80B to 122B and from one GPU to a two-GPU expert-parallel split roughly *thirds* your single-stream speed. Tuning made the big model latency-competitive and its tail latency correct; it could not make cross-node expert parallelism scale with concurrency, because that's physics, not code.

## Rust, specifically

The language was half the point. Working in Rust at this depth — traits for the decode paths, an explicit worker protocol for the multi-box split, buffer-safety reasoning where each per-sequence computation writes into its own slot so a cross-sequence aliasing bug *can't* recur by construction — was a different kind of thinking than the Node/TypeScript/Python/YAML I live in most days. The compiler and the linter are opinionated in a way that pushed me toward designs where the unsafe thing is unrepresentable rather than merely avoided. I came out of it able to read a systems codebase I'd have bounced off a month earlier, and that's the durable win.

I landed the work upstream, and it merged — [PR #101](https://github.com/Avarok-Cybersecurity/atlas/pull/101), "lift max_batch_size=1 under EP=2 via head↔worker slot multiplexing," accepted by the maintainer in early June. It carries the opt-in v2 wire protocol (a `seq_id` preamble so workers route each operation to the right slot), the fused small-batch MoE kernels for two and three concurrent sequences, and the two head-side bugs the work flushed out along the way — logits from different sequences overwriting the same buffer row, and a batched broadcast that deadlocked the cross-node collective by misaligning operation order between ranks. Alongside it went an architecture decision record, [`docs/adr/0011-ep-batched-decode-optimization.md`](https://github.com/Avarok-Cybersecurity/atlas/blob/main/docs/adr/0011-ep-batched-decode-optimization.md), and writing that was the part I'd least expected to value: the ADR's whole thesis is that this batching **eliminates tail latency and head-of-line blocking rather than lifting aggregate throughput** — because the system stays bandwidth-bound — which is exactly the flat-curve lesson from the section above, stated as a decision the next person can inherit instead of a surprise they have to rediscover. The honest frontier past this is a true batched expert-parallel MoE *kernel* for higher concurrency, which the maintainer flagged as the follow-up worth doing; it's genuinely hard, and it's the one thing that would move the aggregate number rather than just the tail.

The two issues I filed here tell the same story this repo keeps telling. [Issue #99](https://github.com/Avarok-Cybersecurity/atlas/issues/99) — the request to lift the concurrency clamp — is closed, resolved by the PR above: I filed it, wrote the fix, and watched it merge, which is about as satisfying as an upstream loop gets. [Issue #94](https://github.com/Avarok-Cybersecurity/atlas/issues/94) is the counterweight: a clean bug report that two of the speculative-decoding flags misbehave on this model under expert parallelism (one loops forever, the other emits garbage, while the path we actually use is fine). It's still open, no maintainer response, no linked fix. A well-scoped feature with a PR attached went all the way through; a standalone bug report with no fix attached is still waiting. Evidence plus a fix travels. Evidence alone waits — same lesson as the scorecard, one repo over.

## What it changed for me

As with a lot of third-party services and apps, I don't trust what the app *says* it did — verify the wire actions, always. Atlas was the same lesson wearing a lab coat. Don't trust the log's acceptance rate, read the counter. Don't assume graphs help, capture one and measure. Don't assume batching scales, run the sweep and watch the line go flat. Every intuition I carried in about where the time goes was wrong until I instrumented the decode step and let the phases tell me: state-space layers were 79% of the step, MoE was the biggest slice inside them, and attention — the thing everyone talks about — was a sixth of the cost.

The arc is parked now; we shut the deployment down to free the hardware for other work, containers preserved. But this was the stretch where the abstractions I'd been standing on turned into code I could read, change, measure, and argue about. Going a level deeper than I'd gone before — into Rust, into the kernels, into the actual numbers — is the part of this whole adventure I'd least want to give back.
